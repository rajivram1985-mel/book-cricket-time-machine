# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Book Cricket Time Machine ŌĆö a browser game reviving Indian schoolyard book cricket (flip to a random page; last digit `0` = out, `1ŌĆō6` = that many runs, `7/8/9` = 1 run), with "what if" duels between real cricket legends. Entirely client-side: vanilla TypeScript + Vite, no framework, no backend, no accounts ŌĆö persistence is a single localStorage blob (`src/storage.ts`), nothing leaves the device. A match is two innings ŌĆö your XI bats, then the rival XI chases; each innings is a spell of 12 balls or 2 wickets (`SPELL` in `src/engine.ts`). The verdict includes a luck report built from per-ball probability snapshots (`expectedRuns`/`outcomeChance`; Classic balls are priced by `classicProbabilities`). The Daily Challenge is a seeded chase (`src/daily.ts`): the rival's innings, players and book are deterministic from the local date, so everyone in the world faces the same target; the player gets one attempt a day, a played-streak, and a Wordle-style emoji share grid.

## Commands

```bash
npm run dev        # Vite dev server
npm test           # vitest run (all tests)
npx vitest run tests/engine.test.ts -t "sums to 1"   # single test by name
npm run build      # tsc type-check + vite build
npm run preview    # serve the production build
npm run voice:generate  # render commentary MP3s via ElevenLabs (needs .env.local, see below)
```

There is no linter configured; `tsc` (strict mode, noUnusedLocals/Parameters) is the only static check and only runs via `npm run build`.

`voice:generate` runs `scripts/generate-voice-clips.ts` via `vite-node` (already a transitive dep of vitest ŌĆö no new package added) so it can import `src/voiceLines.ts`, `src/commentators.ts` and `src/roster.ts` directly as the single source of truth. It loops every persona in `COMMENTATORS` ├Ś every line/name, using each persona's own `voiceId`/`voiceSettings` (not a single global voice). It reads `ELEVENLABS_API_KEY` (required) and optional `ELEVENLABS_MODEL_ID` from a hand-rolled `.env.local` loader (no dotenv dependency; the file is covered by the existing `*.local` gitignore pattern ŌĆö never paste the key into chat or commit it). It's idempotent ŌĆö reruns skip files that already exist ŌĆö and writes into `public/audio/voice/<personaId>/`. Those MP3s are committed like any other static asset (not gitignored): that keeps a git-integrated static host (Vercel/Netlify) able to build and deploy without ever needing the ElevenLabs key as a CI secret. Adding a fifth persona means adding one entry to `COMMENTATORS` and rerunning the script ŌĆö it only synthesizes the new persona's files, since existing ones are skipped.

## Architecture

Two-layer split ŌĆö pure logic vs. DOM ŌĆö is the core invariant:

- **`src/engine.ts`** ŌĆö all game logic as pure functions. Every random draw takes an injectable `Rng` (defaults to `Math.random`) so tests are deterministic. Session-2 additions: `STANCES` (defend/normal/attack multipliers on wicket odds and boundary weights), power play (runs double at scoring time; wicket odds ├Ś2 capped at 60% in Stats, and in Classic the 7/8/9 mercy digits become OUT), the rival chase brain (`chaseStance`, `chaseUsesPowerPlay` ŌĆö pure, so the UI announces intent honestly before the flip), and Gauntlet seeding ratings. **All new `computeProbabilities` parameters default to exact identity** (`'normal'`, `false`) ŌĆö the daily challenge seeds through the default path, and `tests/daily.test.ts` pins Daily #1's full fingerprint as a regression tripwire. Two ball-drawing paths:
  - *Classic mode*: `drawClassic` picks a real page number; outcome follows from its last digit.
  - *Stats mode*: `drawOutcome` samples from `computeProbabilities(bat, bowl, eraAdjusted)`, then `pageForOutcome` reverse-engineers a page whose last digit would have produced that outcome ŌĆö the book metaphor is kept honest by construction, so don't break that pairing.
  - **Strike rate feeds wicket odds too (session 6)** ŌĆö `computeProbabilities` used to let a batsman's strike rate shape only the run distribution, so two batsmen with the same average but wildly different tempos had *identical* dismissal odds ŌĆö a legibility/fairness gap an external review flagged. Now `aggressionRisk(bat)` (a half-weight premium: `1 + (strikeRate/75 - 1) * 0.5`) also multiplies the base wicket odds, so a blaster scores faster but gets out more and a blocker survives longer ŌĆö the intuitive tradeoff. **Anchored at strike rate 75 = exact ├Ś1**, which is why the `normal`/default path stays deterministic for the daily. This change deliberately re-pinned Daily #1 (the seeded rival innings shifted, target 20 ŌåÆ 22) ŌĆö see the regression test's comment; only ~4 days of dailies existed in one small test group, so it was the cheapest moment to accept the re-pin. Don't silently move this formula again.
- **`src/main.ts`** ŌĆö the only file that touches the DOM. Single mutable `State` object with three phases (`home` ŌĆö the nostalgic pavilion/bookshelf screen ŌĆö then `setup` and `play`); setup screen is re-rendered via full `innerHTML` replacement (`render()`), while play-screen ball updates and text inputs are patched by targeted DOM mutation to preserve animations and input focus (see `handleInput`'s "no re-render: don't steal focus" pattern). Events are delegated through `data-action` attributes on one root listener. A daily match is `state.daily !== null`: it skips innings 1 (pre-seeded), always chases, and its verdict has no "Play Again".
  - **Home screen (session 5 redesign)** ŌĆö a three-book shelf (`shelfHtml`/`dailyBookHtml`/`classicBookHtml`/`timeMachineBookHtml`), not the old stacked panels: all three ways to play sit above the fold on every device. The daily book is a clickable `<button>` (gold-outlined, `book-featured`) until played, then becomes a non-interactive muted `<div>` (`book-done`) carrying the stamp/grid/share button ŌĆö and at that point Classic/Time Machine both pick up `book-featured` instead, so the shelf's attention visibly shifts once today's chase is spent (`dailyPlayedToday`). **Watch for class collisions**: `.book-title` already existed for the in-match flip-card caption before this redesign ŌĆö the shelf's heading class is `.book-heading`, deliberately different; don't rename it back.
  - **How-to-play explainer (session 6)** ŌĆö a second `<details class="hero-memory">` block in `homeHtml()`, placed after the rule chips (see below), reusing that same class/`.hero-copy` styling with zero new CSS. Content assumes basic cricket familiarity (no runs/wickets/sixes 101) and only explains what book cricket is plus how this app specifically works: the digit rule, the two-innings match shape (pulls `${eng.SPELL.maxWickets}`/`${eng.SPELL.maxBalls}` rather than hardcoding 2/12), and the pick-a-mode-and-flip flow. It's collapsed by default for everyone, always ŌĆö an early version auto-opened it for first-time visitors, but a 4-paragraph instructional wall as the very first thing on the page undercut the nostalgic hero moment, so it's opt-in via its own summary label instead. The nostalgia block is `open` by default in its place, since it's short and *is* the brand voice.
  - **Home screen above-the-fold pass (session 6)** ŌĆö a simulated first-time-visitor walkthrough found that the nostalgia block, at its original 4-sentence length and `open` by default, pushed the rule chips and the "pick up a book" mode picker entirely off a real phone's first screen ŌĆö a newcomer had to scroll past a paragraph written for someone who already played this in school before reaching anything that applied to them. Fixed by (1) trimming the nostalgia copy to ~3 lines (kept the "6 = Tendulkar, 0 = groaned" emotional beat and the closing tagline, cut the scene-setting filler) and (2) moving `.rules-chips` to sit directly after the nostalgia block, before the collapsed "New here?" toggle ŌĆö the actual rule is now the first thing after the hero headline, not buried below two collapsibles. Verified on a 375├Ś812 viewport: chips, both toggles, and the top of the Daily Challenge card are all visible with zero scrolling now.
  - **Header layout** ŌĆö `.header-toggles` (reduce-motion/sound/voice/commentator picker) is normal in-flow content below the masthead, not absolutely positioned ŌĆö it grew to four rows once the commentator picker shipped and would overlap the tagline at any width if pinned top-right. Don't reintroduce `position: absolute` there.
- **`src/daily.ts`** ŌĆö pure Daily Challenge logic: `mulberry32` seeded PRNG, local-date day keys (never `toISOString` ŌĆö the day rolls at local midnight), and `generateDaily`, which draws players, book and the rival's full innings off one seeded stream. **Draw order is part of the contract** ŌĆö reordering the draws (or editing `DAILY_BOOKS`, the roster order, or `computeProbabilities`) silently changes every past and future challenge. `DAILY_EPOCH_KEY` fixes Daily #1 and must never move once share grids are public. `MIN_DAILY_TARGET` (session 3) is a dud-day floor: if the simulated innings collapses below it, `generateDaily` re-simulates from the *same continuing* rng stream (never re-seeds) via `simulateInnings`, so a day that already clears the floor ŌĆö like Daily #1, target 22 ŌĆö never retries and stays byte-identical. `tests/daily.test.ts` sweeps thousands of day keys to confirm the floor holds; that sweep, plus the Daily #1 pin, must both keep passing before touching this function again. `pickRandomBook` (session 5) is `generateDaily`'s book-pick expression pulled out into its own function ŌĆö `generateDaily` calls it seeded, Classic mode's setup screen calls it unseeded for the default "Surprise me" book. Same `DAILY_BOOKS` pool deliberately, so Classic mode gets the same nostalgic flavor as the daily without a second content list.
- **`src/storage.ts`** ŌĆö the only module that may touch localStorage: one versioned blob (career ledger, daily streaks, luckiest-ever ball, sound pref) behind an injectable `Backing` so tests use a Map and private-browsing mode degrades to in-memory. `beginDailyAttempt` marks the day as played *when the chase starts*, so a mid-match refresh can't buy a second attempt. `Store.resetToDefaults()` (session 5) is the factory reset ŌĆö wired to the footer's "Reset data" link (`resetAllData` in `main.ts`), which confirms via the native `window.confirm()` dialog (no custom modal ŌĆö this app doesn't build one-off UI machinery for a single yes/no), then also clears the service-worker's Cache Storage before reloading, so a reset is a genuinely fresh start, not just wiped stats. **Backup & restore (session 6)** ŌĆö `exportData`/`importData` plus `Store.replaceAll`, wired to footer "Back up"/"Restore". Since there's no backend, this is the *only* way to carry a career to a new phone or survive a cache wipe: Back up downloads the save as a JSON file (entirely client-side ŌĆö the file only ever goes to the user's own downloads); Restore reads a chosen file, and `importData` returns `null` (rather than defaulting) for anything not recognizably a save, so a bad file *warns* instead of silently wiping the current scorebook. Both `loadData` and `importData` share one `normalize()` helper, so every field is coerced/defaulted identically no matter the source. This does not weaken the "no accounts, no network" promise ŌĆö nothing is uploaded.
  - **Daily resume-on-refresh (session 6)** ŌĆö marking the day as played used to be a one-way door: a genuine accidental refresh, tab close, or OS-suspended tab got treated identically to deliberately quitting, permanently costing the day's one attempt ("Walked off mid-chase", no way back in). Fixed by persisting a `DailyProgress` snapshot (`daily.progress` in `SaveData`) after *every* ball via `saveDailyProgress` ŌĆö balls flipped, runs, wickets, momentum, stance, etc. ŌĆö and restoring it in `resumeDaily` (`main.ts`) when the player comes back. **Only ever restores balls already flipped; never draws new ones** ŌĆö `generateDaily(dayKey)` is regenerated fresh (it's pure and deterministic, so this is free), and only the player's own progress through it is persisted, which is why there's no reroll-for-a-better-start exploit. `dailyBookHtml` now renders a clickable "Continue the chase" book instead of the dead "Walked off" card whenever a matching `progress` snapshot exists; the old static "Walked off mid-chase" copy is kept only as a fallback for a `today` with no resumable snapshot (e.g. data saved before this shipped). `completeDailyAttempt` clears `progress` once the match is genuinely finalized. Edge case handled deliberately: if the match-deciding ball already landed but the verdict never got shown before the refresh, `resumeDaily` detects `spellOver` immediately on restore and routes straight to the verdict instead of re-offering a flip (which would otherwise let the last ball be redrawn). `scoreLineText`/`ballsLineText`/`momentumStatusText`/`chipClass`/`pageFaceHtml`/`outcomeBadgeHtml` are shared between the initial render and the live per-ball update specifically so a resumed match and a ball-by-ball-played match can never drift apart ŌĆö don't reintroduce separate formatting logic for either path.
- **`src/roster.ts`** ŌĆö data-only player list. Adding a player requires no simulation changes: bowlers get a `bowling` block, batsmen a `batting` block; all-rounders get both and appear in both pickers. Stats are deliberately approximate format blends tuned for fun.
- **`src/commentary.ts`** ŌĆö phrase templates with `{batsman}`/`{bowler}`/`{page}` placeholders (filled via `fillTemplate`) and a module-level `lastPhrase` guard so no phrase repeats on consecutive balls (tests rely on `resetCommentary()`). `inningsBreakLine(runs, batsman)` (session 6) picks the innings-break clause by a pure `inningsBreakTier(runs)` ŌĆö `duck`/`collapse`/`modest`/`par`/`big` ŌĆö so a 0/2 collapse no longer reads "walks off to schoolyard applause" (that line is now gated to the `big` tier); tier boundaries are unit-tested.
- **`src/audio.ts`** ŌĆö synthesized Web Audio sound cues (flip/runs/boundary/wicket/settle); no audio assets. The sound toggle persists via `storage.ts` like everything else. **The footer promise changed (session 6)**: it used to say "no tracking" absolutely; it now says the scorebook itself never leaves the device and only anonymous, cookie-free usage counts do ŌĆö see the `src/analytics.ts` bullet below before assuming "nothing goes over the network" still holds.
- **`src/analytics.ts` (session 6)** ŌĆö the project's first-ever intentional third-party network call, added deliberately (not silently) after the Play Store launch made "how is this actually being used" an unanswerable question with zero data. Uses [Umami](https://umami.is), a cookie-free analytics service with no persistent cross-visit identifier ŌĆö a "visit" is a same-day, unlinkable hash. **The script is never statically included in `index.html`** ŌĆö `setAnalyticsEnabled(enabled)` injects or removes a `<script id="umami-script">` tag at runtime based on `state.analyticsOn` (backed by `SaveData.prefs.analyticsOn`, default `true`, toggled via the header's "Anonymous usage stats" checkbox). This makes the opt-out real: turning it off removes the script itself, so `window.umami` becomes undefined and every `track()` call ŌĆö which is always wrapped in `window.umami?.track(...)` inside a try/catch ŌĆö silently no-ops, both for the rest of the session and on the next visit. Called once at startup (`setAnalyticsEnabled(state.analyticsOn)` near the bottom of `main.ts`) and again on every toggle and every `importSaveData()` restore (a restored backup's `analyticsOn` value must actually take effect, not just update `state` while the script stays loaded from before).
  - **Exactly seven events, chosen because each answers one named product question, not "track everything"**: `match_started`/`match_finished` (both carry `{ mode: 'classic'|'stats'|'daily'|'challenge' }`, `match_finished` also carries `{ result: 'won'|'lost'|'tied' }`) ŌĆö comparing start vs. finish counts per mode shows both which mode gets played and mid-match abandonment, without a separate event. `daily_share_tapped` (fired in the `copy-daily` click case) ŌĆö is the viral/share loop actually used. `howto_opened` (fired via a **capture-phase** `toggle` listener on `app`, since `<details>`'s native `toggle` event doesn't bubble reliably across browsers ŌĆö `handleHowtoToggle` checks `el.id === 'howto-details' && el.open`) ŌĆö does the onboarding explainer get used at all. `gauntlet_started` (fired once per fresh series in `startSpell`, guarded by the same `if (!state.series)` check that creates the series ŌĆö never fires again for match 2 or 3 of an already-running series) ŌĆö is the deepest feature worth its complexity. `challenge_created` (share button tapped) and `challenge_opened` (a challenge link parsed off the URL) ŌĆö together with `match_started{mode:'challenge'}`/`match_finished` these form the full viral funnel: created ŌåÆ opened ŌåÆ started ŌåÆ finished is the K-factor of the challenge loop (session 7). `match_started` for a resumed Daily chase (`resumeDaily`) is deliberately **not** tracked ŌĆö it's a continuation of an attempt already counted when it began, not a new one.
  - **Events fired before the Umami script finishes loading are queued** (session 7): `challenge_opened` fires at boot, milliseconds after `setAnalyticsEnabled` injects the script tag ŌĆö `window.umami` doesn't exist yet, so `track()` holds such events in a small module-level queue (cap 20, only when the script tag is actually present) and flushes them on the script's `load` event. Without this, every boot-time event would silently vanish. The queue is cleared on opt-out.
  - **The on-device scorebook is never in an event, ever** ŌĆö no player names, no exact scores, no personal totals. Only coarse categories cross the boundary. If you add a new event, it must answer a specific question someone actually asked, and it must pass the same test.
  - `public/privacy.html` and `store-assets/listing.md`'s Data Safety notes were updated in the same commit this shipped ŌĆö **never add a new tracked event without updating both**, since the privacy policy and the Play Store's data-safety disclosure are promises about what currently happens, not just what happened at launch.
  - Untested by design, same as `main.ts` ŌĆö `setAnalyticsEnabled` is pure DOM manipulation (`document.createElement`/`appendChild`/`getElementById`) and the project has no jsdom/happy-dom environment configured; adding one just for this would be disproportionate. Verified in-browser instead: script injects/removes on toggle, events fire with correct data via a spied `window.umami`, and nothing throws when `window.umami` is absent.
- **`src/challenge.ts` (session 7) ŌĆö friend challenges ("beat my score" links), the app's deliberate viral loop.** After any Classic/Time Machine verdict, "ŌÜö’ĖÅ Challenge a friend" packages the player's own batted innings as a URL: the matchup (batId/bowlId), book, era-adjust flag, and score travel in the URL **hash** (`/#challenge=v1.<base64url JSON>`) ŌĆö a hash and not a query string on purpose (never reaches the server or its logs, can't bust the service-worker cache). The friend opening it chases `runs + 1` under the identical odds; their verdict's "ŌÜö’ĖÅ Fire one back" builds a counter-challenge from their own chase. Key invariants:
  - **`decodeChallenge` is deliberately paranoid** ŌĆö every field validated against roster/engine bounds (real roster ids in the correct roles so `.batting!`/`.bowling!` never see null; pages/runs/wickets/balls clamped; stats mode force-overrides the book to `VIRTUAL_BOOK`). Any anomaly ŌåÆ `null` (the link just doesn't land), never a crash. Display-only ball tokens soft-fail to `[]` instead. **No names travel in the link, ever** ŌĆö identity comes from the chat app it's shared through; don't add a name field.
  - **The chase is live RNG** ŌĆö only the target travels, no seed. Structurally a challenge is a daily-style chase: `startChallenge` sets `innings = 2` from ball one, `state.challenge` non-null, and `playerBatting()`/`currentPair()` treat it exactly like `state.daily`. The challenger's innings arrives as tokens with **no luck data** ŌĆö `state.inn1.luck` is `[]`, which is why `recordFinishedMatch` guards `!state.challenge` before touching `inn1.luck[i]` (it would crash) and the challenge verdict shows a chase-only luck line instead of `luckReportHtml()`.
  - **Retries are allowed on purpose** ("took me five goes" is half the fun of a dare); every attempt records as a real match, `challengesWon` increments per winning chase. `pendingChallenge` lives at module level (not in `State`) so `freshSetup`/go-home can't drop an unaccepted dare off the home screen; it's never persisted ŌĆö re-opening the link recreates it.
  - The codec and validation are fully unit-tested (`tests/challenge.test.ts`, includes tamper tests that surgically rewrite wire fields); the DOM flow (landing card, verdict, hashchange) is untested-by-design like the rest of `main.ts` and verified in-browser.

- **FIRM RULE ŌĆö display name vs wire name (session 10, read this before renaming anything in this app).** "Stats mode" is called **Time Machine** on every user-facing surface (the home shelf card, the setup tab, the Play Store listing, the how-to panel) ŌĆö but `Mode = 'classic' | 'stats'` in `types.ts`, `state.mode === 'stats'`, the `data-action="mode-stats"`/`nav-stats"` attributes, the `.book-stats` CSS class, every engine parameter, and the `mode` value inside `match_started`/`match_finished` analytics events are **all still literally `'stats'`, permanently**. If a rename ever needs the *wire* value to change too (e.g. a genuine mode split), that is a breaking migration requiring its own explicit sign-off, not a documentation-driven cleanup.

## Product constraints

- Stats mode is a playful simulation, never a prediction ŌĆö the UI states this disclaimer in multiple places (setup panel, play screen, verdict, footer). Preserve it when touching those surfaces.
- `computeProbabilities` is surfaced verbatim in the in-game "How the odds work" panel; if you change its math, the explanation in `main.ts`'s `oddsBody()` must stay accurate. `oddsBody` renders a legible `<ul class="odds-mods">` of only the *active* factors (matchup, batter tempo from strike rate, settling-in, fatigue, era gap, stance, power play) above the raw percentage table ŌĆö not a wall of prose. When you add or change a modifier in the engine, add/adjust its line here so the panel doesn't quietly lie.
- The player bats innings 1 and every daily chase; the AI rival bats innings 2 of regular matches. Stance buttons and the power-play toggle must never render during the rival's innings ŌĆö the rival's intent line (`aiIntentText`) is the honest mirror of the same pure functions the flip will use.
- **State your batting/bowling role explicitly, never make the player infer it from names (session 6).** Real feedback: a 10-year-old playtester didn't realize he was bowling in innings 2 ŌĆö the only signals were the matchup names and the absent stance buttons, both of which read fine to an adult who already knows the game but say nothing explicit to a kid. `playHtml`'s `innings-banner` now always states `You bat` / `You bowl` up front (driven by `playerBatting()`), and the fighter card matching the player's own role gets a `┬Ę you` suffix. Don't regress this back to name-only banners even if it reads as redundant to an adult ŌĆö the target audience spans a wide age range and the redundancy is the point.
- One power play per innings per side (`ppUsed` resets in `startChase`/`startSpell`/`startDaily`). A doubled ball stores the *drawn* outcome (`Ball.doubled` flag, `├Ś2` token suffix); doubling is applied at scoring/display time so the page metaphor, luck pricing and share grid stay consistent.
- `#flip-btn` is `position: sticky; bottom: 14px` (session 5) ŌĆö it must stay reachable without scrolling on any screen height, and stay put as `#ball-log` grows a chip taller every single ball (without this it would slide further down the page each flip, not just at match start). Don't remove the sticky positioning to "simplify" the CSS.
- `.verdict` has `max-height: calc(100vh - 32px); overflow-y: auto` (session 5) ŌĆö a Gauntlet-conclusion verdict (series line + next-opponent teaser + record notes + luck report + disclaimer + 3 action buttons) is taller than most viewports, and `.overlay` itself doesn't scroll. Without the modal's own internal scroll, both the heading and the action buttons become completely unreachable ŌĆö confirmed by reproducing it at a 600px-tall viewport before the fix existed. Keep this whenever adding more content to any verdict.
- Respect the `reduceMotion` state flag: animations gate on it and durations drop to 0.

## Google Play (TWA)

The Play build is a Bubblewrap-generated **Trusted Web Activity** wrapping the live Netlify site ŌĆö no forked codebase, deploys update the app instantly. Store-critical files in this repo: `public/.well-known/assetlinks.json` (fingerprint placeholder must become the **Play App Signing** key SHA-256 from the Console, *not* the upload key ŌĆö the app shows a browser URL bar until it matches), `public/privacy.html` (required listing URL), `public/icons/icon-*-maskable.png` (full-bleed, `purpose: maskable` ŌĆö the framed tile icons get their corners clipped by Android's adaptive-icon mask, so don't reuse them for Android/Play surfaces), and `store-assets/` (feature graphic + listing copy + SVG sources). Package name `com.bookcricket.timemachine` is permanent once published even if the domain changes. See README "Google Play (TWA)" for the full pipeline.
