# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Book Cricket Time Machine ‚Äî a browser game reviving Indian schoolyard book cricket (flip to a random page; last digit `0` = out, `1‚Äì6` = that many runs, `7/8/9` = 1 run), with "what if" duels between real cricket legends. Entirely client-side: vanilla TypeScript + Vite, no framework, no backend, no accounts ‚Äî persistence is a single localStorage blob (`src/storage.ts`), nothing leaves the device. A match is two innings ‚Äî your XI bats, then the rival XI chases; each innings is a spell of 12 balls or 2 wickets (`SPELL` in `src/engine.ts`). The verdict includes a luck report built from per-ball probability snapshots (`expectedRuns`/`outcomeChance`; Classic balls are priced by `classicProbabilities`). The Daily Challenge is a seeded chase (`src/daily.ts`): the rival's innings, players and book are deterministic from the local date, so everyone in the world faces the same target; the player gets one attempt a day, a played-streak, and a Wordle-style emoji share grid.

## Commands

```bash
npm run dev        # Vite dev server
npm test           # vitest run (all tests)
npx vitest run tests/engine.test.ts -t "sums to 1"   # single test by name
npm run build      # tsc type-check + vite build
npm run preview    # serve the production build
npm run voice:generate  # render commentary MP3s via ElevenLabs (needs .env.local, see below)
```

There is no linter configured; `tsc` (strict mode, noUnusedLocals/Parameters) is the only static check and only runs via `npm run build`.

`voice:generate` runs `scripts/generate-voice-clips.ts` via `vite-node` (already a transitive dep of vitest ‚Äî no new package added) so it can import `src/voiceLines.ts`, `src/commentators.ts` and `src/roster.ts` directly as the single source of truth. It loops every persona in `COMMENTATORS` √ó every line/name, using each persona's own `voiceId`/`voiceSettings` (not a single global voice). It reads `ELEVENLABS_API_KEY` (required) and optional `ELEVENLABS_MODEL_ID` from a hand-rolled `.env.local` loader (no dotenv dependency; the file is covered by the existing `*.local` gitignore pattern ‚Äî never paste the key into chat or commit it). It's idempotent ‚Äî reruns skip files that already exist ‚Äî and writes into `public/audio/voice/<personaId>/`. Those MP3s are committed like any other static asset (not gitignored): that keeps a git-integrated static host (Vercel/Netlify) able to build and deploy without ever needing the ElevenLabs key as a CI secret. Adding a fifth persona means adding one entry to `COMMENTATORS` and rerunning the script ‚Äî it only synthesizes the new persona's files, since existing ones are skipped.

## Architecture

Two-layer split ‚Äî pure logic vs. DOM ‚Äî is the core invariant:

- **`src/engine.ts`** ‚Äî all game logic as pure functions. Every random draw takes an injectable `Rng` (defaults to `Math.random`) so tests are deterministic. Session-2 additions: `STANCES` (defend/normal/attack multipliers on wicket odds and boundary weights), power play (runs double at scoring time; wicket odds √ó2 capped at 60% in Stats, and in Classic the 7/8/9 mercy digits become OUT), the rival chase brain (`chaseStance`, `chaseUsesPowerPlay` ‚Äî pure, so the UI announces intent honestly before the flip), and Gauntlet seeding ratings. **All new `computeProbabilities` parameters default to exact identity** (`'normal'`, `false`) ‚Äî the daily challenge seeds through the default path, and `tests/daily.test.ts` pins Daily #1's full fingerprint as a regression tripwire. Two ball-drawing paths:
  - *Classic mode*: `drawClassic` picks a real page number; outcome follows from its last digit.
  - *Stats mode*: `drawOutcome` samples from `computeProbabilities(bat, bowl, eraAdjusted)`, then `pageForOutcome` reverse-engineers a page whose last digit would have produced that outcome ‚Äî the book metaphor is kept honest by construction, so don't break that pairing.
- **`src/main.ts`** ‚Äî the only file that touches the DOM. Single mutable `State` object with three phases (`home` ‚Äî the nostalgic pavilion/bookshelf screen ‚Äî then `setup` and `play`); setup screen is re-rendered via full `innerHTML` replacement (`render()`), while play-screen ball updates and text inputs are patched by targeted DOM mutation to preserve animations and input focus (see `handleInput`'s "no re-render: don't steal focus" pattern). Events are delegated through `data-action` attributes on one root listener. A daily match is `state.daily !== null`: it skips innings 1 (pre-seeded), always chases, and its verdict has no "Play Again".
  - **Home screen (session 5 redesign)** ‚Äî a three-book shelf (`shelfHtml`/`dailyBookHtml`/`classicBookHtml`/`timeMachineBookHtml`), not the old stacked panels: all three ways to play sit above the fold on every device. The daily book is a clickable `<button>` (gold-outlined, `book-featured`) until played, then becomes a non-interactive muted `<div>` (`book-done`) carrying the stamp/grid/share button ‚Äî and at that point Classic/Time Machine both pick up `book-featured` instead, so the shelf's attention visibly shifts once today's chase is spent (`dailyPlayedToday`). **Watch for class collisions**: `.book-title` already existed for the in-match flip-card caption before this redesign ‚Äî the shelf's heading class is `.book-heading`, deliberately different; don't rename it back.
  - **How-to-play explainer (session 6)** ‚Äî a second `<details class="hero-memory">` block in `homeHtml()`, placed before the "Remember the last bench?" nostalgia block, reusing that same class/`.hero-copy` styling with zero new CSS. Content assumes basic cricket familiarity (no runs/wickets/sixes 101) and only explains what book cricket is plus how this app specifically works: the digit rule, the two-innings match shape (pulls `${eng.SPELL.maxWickets}`/`${eng.SPELL.maxBalls}` rather than hardcoding 2/12), and the pick-a-mode-and-flip flow. It defaults `open` when `store.data.career.matches === 0` (first-time visitor) and stays collapsed thereafter ‚Äî same signal `ledgerStripHtml()` already uses to distinguish new vs. returning players.
  - **Header layout** ‚Äî `.header-toggles` (reduce-motion/sound/voice/commentator picker) is normal in-flow content below the masthead, not absolutely positioned ‚Äî it grew to four rows once the commentator picker shipped and would overlap the tagline at any width if pinned top-right. Don't reintroduce `position: absolute` there.
- **`src/daily.ts`** ‚Äî pure Daily Challenge logic: `mulberry32` seeded PRNG, local-date day keys (never `toISOString` ‚Äî the day rolls at local midnight), and `generateDaily`, which draws players, book and the rival's full innings off one seeded stream. **Draw order is part of the contract** ‚Äî reordering the draws (or editing `DAILY_BOOKS`, the roster order, or `computeProbabilities`) silently changes every past and future challenge. `DAILY_EPOCH_KEY` fixes Daily #1 and must never move once share grids are public. `MIN_DAILY_TARGET` (session 3) is a dud-day floor: if the simulated innings collapses below it, `generateDaily` re-simulates from the *same continuing* rng stream (never re-seeds) via `simulateInnings`, so a day that already clears the floor ‚Äî like Daily #1, target 20 ‚Äî never retries and stays byte-identical. `tests/daily.test.ts` sweeps thousands of day keys to confirm the floor holds; that sweep, plus the Daily #1 pin, must both keep passing before touching this function again. `pickRandomBook` (session 5) is `generateDaily`'s book-pick expression pulled out into its own function ‚Äî `generateDaily` calls it seeded, Classic mode's setup screen calls it unseeded for the default "Surprise me" book. Same `DAILY_BOOKS` pool deliberately, so Classic mode gets the same nostalgic flavor as the daily without a second content list.
- **`src/storage.ts`** ‚Äî the only module that may touch localStorage: one versioned blob (career ledger, daily streaks, luckiest-ever ball, sound pref) behind an injectable `Backing` so tests use a Map and private-browsing mode degrades to in-memory. `beginDailyAttempt` marks the day as played *when the chase starts*, so a mid-match refresh can't buy a second attempt. `Store.resetToDefaults()` (session 5) is the factory reset ‚Äî wired to the footer's "Reset data" link (`resetAllData` in `main.ts`), which confirms via the native `window.confirm()` dialog (no custom modal ‚Äî this app doesn't build one-off UI machinery for a single yes/no), then also clears the service-worker's Cache Storage before reloading, so a reset is a genuinely fresh start, not just wiped stats.
- **`src/roster.ts`** ‚Äî data-only player list. Adding a player requires no simulation changes: bowlers get a `bowling` block, batsmen a `batting` block; all-rounders get both and appear in both pickers. Stats are deliberately approximate format blends tuned for fun.
- **`src/commentary.ts`** ‚Äî phrase templates with `{batsman}`/`{bowler}`/`{page}` placeholders (filled via `fillTemplate`) and a module-level `lastPhrase` guard so no phrase repeats on consecutive balls (tests rely on `resetCommentary()`).
- **`src/audio.ts`** ‚Äî synthesized Web Audio sound cues (flip/runs/boundary/wicket); no audio assets. The sound toggle persists via `storage.ts` like everything else. The footer promise is "your scorebook lives only on this device ‚Äî no accounts, no tracking": localStorage is fine, but nothing may ever be sent over the network.
- **`src/voiceLines.ts` / `src/commentators.ts` / `src/voice.ts`** ‚Äî pre-generated commentary, voiced on wickets, fours, sixes, power plays and verdicts only (never singles/twos/threes/fives ‚Äî see `resolveBallMoment`'s doc comment). `voiceLines.ts` is name-free script text (dynamic names stay in `commentary.ts`'s on-screen text). `commentators.ts` defines four **original commentator personas** (Enthusiast, Deadpan, Analyst, Showman) ‚Äî each an ElevenLabs stock/library voice pushed toward a character via `voiceSettings`, never a clone or impersonation of a real, named commentator; all four personas read the identical script, so the character comes entirely from the voice actor and delivery tuning. **Do not add a persona built from a real person's cloned voice** ‚Äî this was explicitly requested and declined (consent/right-of-publicity risk, worse for anyone deceased); stock/licensed library voices with original character framing are the line. `voice.ts` has two layers: pure resolvers (`resolveBallMoment`, `resolveMatchMoment`, unit-tested, and deliberately mirror `commentaryFor`'s priority order ‚Äî a `tests/voice.test.ts` case cross-checks the two never disagree) and untested browser playback (`playMomentVoice`/`playNameCallout`, both take a `personaId`) that fails silently on a 404. **A plain six has a 35% chance of playing just the batsman's name instead of a full line** (`revealBall` in `main.ts`) ‚Äî that's realistic (commentators really do just shout a name on a huge six). **A wicket never does this** ‚Äî real commentary always signals the dismissal itself ("OUT!", "Gone!"), so a bare bowler's name with nothing else doesn't read as a wicket; wickets always get a full reaction line. Don't add a symmetric bare-name shortcut for wickets if this comes up again. **`earlyDuck` (session 6)** is a fifth wicket flavor: a plain (not attacking, not power-play) dismissal within the player's first 3 balls faced this innings (`state.earlyDuckThisMatch`, set in `revealBall`, read once at verdict time ‚Äî survives innings 1‚Üí2 so it isn't lost by the time the match ends). Priority in both `resolveBallMoment` and `commentaryFor` is doubled > attacking > earlyDuck > plain wicket ‚Äî a *chosen* risk always keeps its own more specific story. This was a deliberate answer to real feedback (a friend got out early and joked about it on WhatsApp): rather than secretly softening Classic mode's odds for beginners ‚Äî which would have broken its "pure page-flip luck, transparent rules" promise ‚Äî the fix is to lean into an early dismissal as a badge (`career.earlyDucks` counter, shown on the ledger strip once >0) rather than hide it. **Do not add hidden difficulty adjustment to Classic mode** if this comes up again; the game's stated odds must stay the same for every player, always. **The game must be fully playable with zero audio files present** ‚Äî clips are static MP3s generated by `scripts/generate-voice-clips.ts` (`npm run voice:generate`, needs `ELEVENLABS_API_KEY` in a gitignored `.env.local`; never call this API from the shipped app itself, or the key would leak to every player), one set per persona under `public/audio/voice/<personaId>/`.
- **`src/avatar.ts`** ‚Äî generated inline-SVG avatars (initials + emoji + country gradient); no photo assets.
- **PWA shell (session 3)** ‚Äî `public/manifest.webmanifest`, `public/icons/*.png` (generated once via a throwaway `sharp` install ‚Äî `npm install --no-save sharp`, never a persisted dependency ‚Äî from the SVG source described in the README; regenerate the same way if the mark changes), and `public/sw.js`, a hand-rolled service worker (no build plugin, no new dependency): cache-first-with-background-refresh for same-origin GET requests, a small explicit precache list (`/`, `/manifest.webmanifest`), registered unconditionally in `main.ts`. Verify PWA behavior against `npm run serve:pwa` (builds, then serves `dist/` on :4300) ‚Äî the raw `npm run dev` server also registers the SW, which can cause stale-content confusion while iterating; unregister it in devtools if that happens.
- **`shareOrCopy` in `main.ts`** (session 3) ‚Äî tries `navigator.share` first (native share sheet, mobile), swallows `AbortError` (user cancelled) silently, falls back to the existing `copyToClipboard` for everything else including desktop browsers without the API. `CAN_SHARE` is detected once at module load, driving the button label (`üì§ Share ‚Ä¶` vs `üìã Copy ‚Ä¶`).

Tests (`tests/engine.test.ts`, `tests/daily.test.ts`, `tests/storage.test.ts`, `tests/voice.test.ts`, `tests/commentators.test.ts`, vitest) cover only the pure layer ‚Äî engine math, commentary non-repetition, daily determinism, storage recorders, voice-moment resolution, and the persona manifest's sanity (unique ids, valid `voiceSettings` ranges). Keep new game logic in `engine.ts`/`daily.ts`/`voice.ts`'s pure half so it stays testable; `main.ts` (and `voice.ts`'s playback half) is untested by design.

`design-system/` holds standalone HTML preview cards (tokens, components) for the claude.ai/design pane, each tagged with a `@dsCard` marker ‚Äî they are reference artifacts, not part of the build.

## Product constraints

- Stats mode is a playful simulation, never a prediction ‚Äî the UI states this disclaimer in multiple places (setup panel, play screen, verdict, footer). Preserve it when touching those surfaces.
- `computeProbabilities` is surfaced verbatim in the in-game "How the odds work" panel; if you change its math, the explanation text in `main.ts`'s `oddsPanel()` must stay accurate ‚Äî including the stance/power-play multiplier lines in `oddsBody()`.
- The player bats innings 1 and every daily chase; the AI rival bats innings 2 of regular matches. Stance buttons and the power-play toggle must never render during the rival's innings ‚Äî the rival's intent line (`aiIntentText`) is the honest mirror of the same pure functions the flip will use.
- One power play per innings per side (`ppUsed` resets in `startChase`/`startSpell`/`startDaily`). A doubled ball stores the *drawn* outcome (`Ball.doubled` flag, `√ó2` token suffix); doubling is applied at scoring/display time so the page metaphor, luck pricing and share grid stay consistent.
- `#flip-btn` is `position: sticky; bottom: 14px` (session 5) ‚Äî it must stay reachable without scrolling on any screen height, and stay put as `#ball-log` grows a chip taller every single ball (without this it would slide further down the page each flip, not just at match start). Don't remove the sticky positioning to "simplify" the CSS.
- `.verdict` has `max-height: calc(100vh - 32px); overflow-y: auto` (session 5) ‚Äî a Gauntlet-conclusion verdict (series line + next-opponent teaser + record notes + luck report + disclaimer + 3 action buttons) is taller than most viewports, and `.overlay` itself doesn't scroll. Without the modal's own internal scroll, both the heading and the action buttons become completely unreachable ‚Äî confirmed by reproducing it at a 600px-tall viewport before the fix existed. Keep this whenever adding more content to any verdict.
- Respect the `reduceMotion` state flag: animations gate on it and durations drop to 0.
