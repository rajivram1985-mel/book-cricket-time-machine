# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Book Cricket Time Machine ‚Äî a browser game reviving Indian schoolyard book cricket (flip to a random page; last digit `0` = out, `1‚Äì6` = that many runs, `7/8/9` = 1 run), with "what if" duels between real cricket legends. Entirely client-side: vanilla TypeScript + Vite, no framework, no backend, no accounts ‚Äî persistence is a single localStorage blob (`src/storage.ts`), nothing leaves the device. A match is two innings ‚Äî your XI bats, then the rival XI chases; each innings is a spell of 12 balls or 2 wickets (`SPELL` in `src/engine.ts`). The verdict includes a luck report built from per-ball probability snapshots (`expectedRuns`/`outcomeChance`; Classic balls are priced by `classicProbabilities`). The Daily Challenge is a seeded chase (`src/daily.ts`): the rival's innings, players and book are deterministic from the local date, so everyone in the world faces the same target; the player gets one attempt a day, a played-streak, and a Wordle-style emoji share grid.

## Commands

```bash
npm run dev        # Vite dev server
npm test           # vitest run (all tests)
npx vitest run tests/engine.test.ts -t "sums to 1"   # single test by name
npm run build      # tsc type-check + vite build
npm run preview    # serve the production build
npm run voice:generate  # render commentary MP3s via ElevenLabs (needs .env.local, see below)
```

There is no linter configured; `tsc` (strict mode, noUnusedLocals/Parameters) is the only static check and only runs via `npm run build`.

`voice:generate` runs `scripts/generate-voice-clips.ts` via `vite-node` (already a transitive dep of vitest ‚Äî no new package added) so it can import `src/voiceLines.ts`, `src/commentators.ts` and `src/roster.ts` directly as the single source of truth. It loops every persona in `COMMENTATORS` √ó every line/name, using each persona's own `voiceId`/`voiceSettings` (not a single global voice). It reads `ELEVENLABS_API_KEY` (required) and optional `ELEVENLABS_MODEL_ID` from a hand-rolled `.env.local` loader (no dotenv dependency; the file is covered by the existing `*.local` gitignore pattern ‚Äî never paste the key into chat or commit it). It's idempotent ‚Äî reruns skip files that already exist ‚Äî and writes into `public/audio/voice/<personaId>/`. Those MP3s are committed like any other static asset (not gitignored): that keeps a git-integrated static host (Vercel/Netlify) able to build and deploy without ever needing the ElevenLabs key as a CI secret. Adding a fifth persona means adding one entry to `COMMENTATORS` and rerunning the script ‚Äî it only synthesizes the new persona's files, since existing ones are skipped.

## Architecture

Two-layer split ‚Äî pure logic vs. DOM ‚Äî is the core invariant:

- **`src/engine.ts`** ‚Äî all game logic as pure functions. Every random draw takes an injectable `Rng` (defaults to `Math.random`) so tests are deterministic. Session-2 additions: `STANCES` (defend/normal/attack multipliers on wicket odds and boundary weights), power play (runs double at scoring time; wicket odds √ó2 capped at 60% in Stats, and in Classic the 7/8/9 mercy digits become OUT), the rival chase brain (`chaseStance`, `chaseUsesPowerPlay` ‚Äî pure, so the UI announces intent honestly before the flip), and Gauntlet seeding ratings. **All new `computeProbabilities` parameters default to exact identity** (`'normal'`, `false`) ‚Äî the daily challenge seeds through the default path, and `tests/daily.test.ts` pins Daily #1's full fingerprint as a regression tripwire. Two ball-drawing paths:
  - *Classic mode*: `drawClassic` picks a real page number; outcome follows from its last digit.
  - *Stats mode*: `drawOutcome` samples from `computeProbabilities(bat, bowl, eraAdjusted)`, then `pageForOutcome` reverse-engineers a page whose last digit would have produced that outcome ‚Äî the book metaphor is kept honest by construction, so don't break that pairing.
  - **Strike rate feeds wicket odds too (session 6)** ‚Äî `computeProbabilities` used to let a batsman's strike rate shape only the run distribution, so two batsmen with the same average but wildly different tempos had *identical* dismissal odds ‚Äî a legibility/fairness gap an external review flagged. Now `aggressionRisk(bat)` (a half-weight premium: `1 + (strikeRate/75 - 1) * 0.5`) also multiplies the base wicket odds, so a blaster scores faster but gets out more and a blocker survives longer ‚Äî the intuitive tradeoff. **Anchored at strike rate 75 = exact √ó1**, which is why the `normal`/default path stays deterministic for the daily. This change deliberately re-pinned Daily #1 (the seeded rival innings shifted, target 20 ‚Üí 22) ‚Äî see the regression test's comment; only ~4 days of dailies existed in one small test group, so it was the cheapest moment to accept the re-pin. Don't silently move this formula again.
- **`src/main.ts`** ‚Äî the only file that touches the DOM. Single mutable `State` object with three phases (`home` ‚Äî the nostalgic pavilion/bookshelf screen ‚Äî then `setup` and `play`); setup screen is re-rendered via full `innerHTML` replacement (`render()`), while play-screen ball updates and text inputs are patched by targeted DOM mutation to preserve animations and input focus (see `handleInput`'s "no re-render: don't steal focus" pattern). Events are delegated through `data-action` attributes on one root listener. A daily match is `state.daily !== null`: it skips innings 1 (pre-seeded), always chases, and its verdict has no "Play Again".
  - **Home screen (session 5 redesign)** ‚Äî a three-book shelf (`shelfHtml`/`dailyBookHtml`/`classicBookHtml`/`timeMachineBookHtml`), not the old stacked panels: all three ways to play sit above the fold on every device. The daily book is a clickable `<button>` (gold-outlined, `book-featured`) until played, then becomes a non-interactive muted `<div>` (`book-done`) carrying the stamp/grid/share button ‚Äî and at that point Classic/Time Machine both pick up `book-featured` instead, so the shelf's attention visibly shifts once today's chase is spent (`dailyPlayedToday`). **Watch for class collisions**: `.book-title` already existed for the in-match flip-card caption before this redesign ‚Äî the shelf's heading class is `.book-heading`, deliberately different; don't rename it back.
  - **How-to-play explainer (session 6)** ‚Äî a second `<details class="hero-memory">` block in `homeHtml()`, placed after the rule chips (see below), reusing that same class/`.hero-copy` styling with zero new CSS. Content assumes basic cricket familiarity (no runs/wickets/sixes 101) and only explains what book cricket is plus how this app specifically works: the digit rule, the two-innings match shape (pulls `${eng.SPELL.maxWickets}`/`${eng.SPELL.maxBalls}` rather than hardcoding 2/12), and the pick-a-mode-and-flip flow. It's collapsed by default for everyone, always ‚Äî an early version auto-opened it for first-time visitors, but a 4-paragraph instructional wall as the very first thing on the page undercut the nostalgic hero moment, so it's opt-in via its own summary label instead. The nostalgia block is `open` by default in its place, since it's short and *is* the brand voice.
  - **Home screen above-the-fold pass (session 6)** ‚Äî a simulated first-time-visitor walkthrough found that the nostalgia block, at its original 4-sentence length and `open` by default, pushed the rule chips and the "pick up a book" mode picker entirely off a real phone's first screen ‚Äî a newcomer had to scroll past a paragraph written for someone who already played this in school before reaching anything that applied to them. Fixed by (1) trimming the nostalgia copy to ~3 lines (kept the "6 = Tendulkar, 0 = groaned" emotional beat and the closing tagline, cut the scene-setting filler) and (2) moving `.rules-chips` to sit directly after the nostalgia block, before the collapsed "New here?" toggle ‚Äî the actual rule is now the first thing after the hero headline, not buried below two collapsibles. Verified on a 375√ó812 viewport: chips, both toggles, and the top of the Daily Challenge card are all visible with zero scrolling now.
  - **Header layout** ‚Äî `.header-toggles` (reduce-motion/sound/voice/commentator picker) is normal in-flow content below the masthead, not absolutely positioned ‚Äî it grew to four rows once the commentator picker shipped and would overlap the tagline at any width if pinned top-right. Don't reintroduce `position: absolute` there.
- **`src/daily.ts`** ‚Äî pure Daily Challenge logic: `mulberry32` seeded PRNG, local-date day keys (never `toISOString` ‚Äî the day rolls at local midnight), and `generateDaily`, which draws players, book and the rival's full innings off one seeded stream. **Draw order is part of the contract** ‚Äî reordering the draws (or editing `DAILY_BOOKS`, the roster order, or `computeProbabilities`) silently changes every past and future challenge. `DAILY_EPOCH_KEY` fixes Daily #1 and must never move once share grids are public. `MIN_DAILY_TARGET` (session 3) is a dud-day floor: if the simulated innings collapses below it, `generateDaily` re-simulates from the *same continuing* rng stream (never re-seeds) via `simulateInnings`, so a day that already clears the floor ‚Äî like Daily #1, target 22 ‚Äî never retries and stays byte-identical. `tests/daily.test.ts` sweeps thousands of day keys to confirm the floor holds; that sweep, plus the Daily #1 pin, must both keep passing before touching this function again. `pickRandomBook` (session 5) is `generateDaily`'s book-pick expression pulled out into its own function ‚Äî `generateDaily` calls it seeded, Classic mode's setup screen calls it unseeded for the default "Surprise me" book. Same `DAILY_BOOKS` pool deliberately, so Classic mode gets the same nostalgic flavor as the daily without a second content list.
- **`src/storage.ts`** ‚Äî the only module that may touch localStorage: one versioned blob (career ledger, daily streaks, luckiest-ever ball, sound pref) behind an injectable `Backing` so tests use a Map and private-browsing mode degrades to in-memory. `beginDailyAttempt` marks the day as played *when the chase starts*, so a mid-match refresh can't buy a second attempt. `Store.resetToDefaults()` (session 5) is the factory reset ‚Äî wired to the footer's "Reset data" link (`resetAllData` in `main.ts`), which confirms via the native `window.confirm()` dialog (no custom modal ‚Äî this app doesn't build one-off UI machinery for a single yes/no), then also clears the service-worker's Cache Storage before reloading, so a reset is a genuinely fresh start, not just wiped stats. **Backup & restore (session 6)** ‚Äî `exportData`/`importData` plus `Store.replaceAll`, wired to footer "Back up"/"Restore". Since there's no backend, this is the *only* way to carry a career to a new phone or survive a cache wipe: Back up downloads the save as a JSON file (entirely client-side ‚Äî the file only ever goes to the user's own downloads); Restore reads a chosen file, and `importData` returns `null` (rather than defaulting) for anything not recognizably a save, so a bad file *warns* instead of silently wiping the current scorebook. Both `loadData` and `importData` share one `normalize()` helper, so every field is coerced/defaulted identically no matter the source. This does not weaken the "no accounts, no network" promise ‚Äî nothing is uploaded.
  - **Daily resume-on-refresh (session 6)** ‚Äî marking the day as played used to be a one-way door: a genuine accidental refresh, tab close, or OS-suspended tab got treated identically to deliberately quitting, permanently costing the day's one attempt ("Walked off mid-chase", no way back in). Fixed by persisting a `DailyProgress` snapshot (`daily.progress` in `SaveData`) after *every* ball via `saveDailyProgress` ‚Äî balls flipped, runs, wickets, momentum, stance, etc. ‚Äî and restoring it in `resumeDaily` (`main.ts`) when the player comes back. **Only ever restores balls already flipped; never draws new ones** ‚Äî `generateDaily(dayKey)` is regenerated fresh (it's pure and deterministic, so this is free), and only the player's own progress through it is persisted, which is why there's no reroll-for-a-better-start exploit. `dailyBookHtml` now renders a clickable "Continue the chase" book instead of the dead "Walked off" card whenever a matching `progress` snapshot exists; the old static "Walked off mid-chase" copy is kept only as a fallback for a `today` with no resumable snapshot (e.g. data saved before this shipped). `completeDailyAttempt` clears `progress` once the match is genuinely finalized. Edge case handled deliberately: if the match-deciding ball already landed but the verdict never got shown before the refresh, `resumeDaily` detects `spellOver` immediately on restore and routes straight to the verdict instead of re-offering a flip (which would otherwise let the last ball be redrawn). `scoreLineText`/`ballsLineText`/`momentumStatusText`/`chipClass`/`pageFaceHtml`/`outcomeBadgeHtml` are shared between the initial render and the live per-ball update specifically so a resumed match and a ball-by-ball-played match can never drift apart ‚Äî don't reintroduce separate formatting logic for either path.
- **`src/roster.ts`** ‚Äî data-only player list. Adding a player requires no simulation changes: bowlers get a `bowling` block, batsmen a `batting` block; all-rounders get both and appear in both pickers. Stats are deliberately approximate format blends tuned for fun.
- **`src/commentary.ts`** ‚Äî phrase templates with `{batsman}`/`{bowler}`/`{page}` placeholders (filled via `fillTemplate`) and a module-level `lastPhrase` guard so no phrase repeats on consecutive balls (tests rely on `resetCommentary()`). `inningsBreakLine(runs, batsman)` (session 6) picks the innings-break clause by a pure `inningsBreakTier(runs)` ‚Äî `duck`/`collapse`/`modest`/`par`/`big` ‚Äî so a 0/2 collapse no longer reads "walks off to schoolyard applause" (that line is now gated to the `big` tier); tier boundaries are unit-tested.
- **`src/audio.ts`** ‚Äî synthesized Web Audio sound cues (flip/runs/boundary/wicket/settle); no audio assets. The sound toggle persists via `storage.ts` like everything else. **The footer promise changed (session 6)**: it used to say "no tracking" absolutely; it now says the scorebook itself never leaves the device and only anonymous, cookie-free usage counts do ‚Äî see the `src/analytics.ts` bullet below before assuming "nothing goes over the network" still holds.
- **`src/analytics.ts` (session 6)** ‚Äî the project's first-ever intentional third-party network call, added deliberately (not silently) after the Play Store launch made "how is this actually being used" an unanswerable question with zero data. Uses [Umami](https://umami.is), a cookie-free analytics service with no persistent cross-visit identifier ‚Äî a "visit" is a same-day, unlinkable hash. **The script is never statically included in `index.html`** ‚Äî `setAnalyticsEnabled(enabled)` injects or removes a `<script id="umami-script">` tag at runtime based on `state.analyticsOn` (backed by `SaveData.prefs.analyticsOn`, default `true`, toggled via the header's "Anonymous usage stats" checkbox). This makes the opt-out real: turning it off removes the script itself, so `window.umami` becomes undefined and every `track()` call ‚Äî which is always wrapped in `window.umami?.track(...)` inside a try/catch ‚Äî silently no-ops, both for the rest of the session and on the next visit. Called once at startup (`setAnalyticsEnabled(state.analyticsOn)` near the bottom of `main.ts`) and again on every toggle and every `importSaveData()` restore (a restored backup's `analyticsOn` value must actually take effect, not just update `state` while the script stays loaded from before).
  - **Exactly seven events, chosen because each answers one named product question, not "track everything"**: `match_started`/`match_finished` (both carry `{ mode: 'classic'|'stats'|'daily'|'challenge' }`, `match_finished` also carries `{ result: 'won'|'lost'|'tied' }`) ‚Äî comparing start vs. finish counts per mode shows both which mode gets played and mid-match abandonment, without a separate event. `daily_share_tapped` (fired in the `copy-daily` click case) ‚Äî is the viral/share loop actually used. `howto_opened` (fired via a **capture-phase** `toggle` listener on `app`, since `<details>`'s native `toggle` event doesn't bubble reliably across browsers ‚Äî `handleHowtoToggle` checks `el.id === 'howto-details' && el.open`) ‚Äî does the onboarding explainer get used at all. `gauntlet_started` (fired once per fresh series in `startSpell`, guarded by the same `if (!state.series)` check that creates the series ‚Äî never fires again for match 2 or 3 of an already-running series) ‚Äî is the deepest feature worth its complexity. `challenge_created` (share button tapped) and `challenge_opened` (a challenge link parsed off the URL) ‚Äî together with `match_started{mode:'challenge'}`/`match_finished` these form the full viral funnel: created ‚Üí opened ‚Üí started ‚Üí finished is the K-factor of the challenge loop (session 7). `match_started` for a resumed Daily chase (`resumeDaily`) is deliberately **not** tracked ‚Äî it's a continuation of an attempt already counted when it began, not a new one.
  - **Events fired before the Umami script finishes loading are queued** (session 7): `challenge_opened` fires at boot, milliseconds after `setAnalyticsEnabled` injects the script tag ‚Äî `window.umami` doesn't exist yet, so `track()` holds such events in a small module-level queue (cap 20, only when the script tag is actually present) and flushes them on the script's `load` event. Without this, every boot-time event would silently vanish. The queue is cleared on opt-out.
  - **The on-device scorebook is never in an event, ever** ‚Äî no player names, no exact scores, no personal totals. Only coarse categories cross the boundary. If you add a new event, it must answer a specific question someone actually asked, and it must pass the same test.
  - `public/privacy.html` and `store-assets/listing.md`'s Data Safety notes were updated in the same commit this shipped ‚Äî **never add a new tracked event without updating both**, since the privacy policy and the Play Store's data-safety disclosure are promises about what currently happens, not just what happened at launch.
  - Untested by design, same as `main.ts` ‚Äî `setAnalyticsEnabled` is pure DOM manipulation (`document.createElement`/`appendChild`/`getElementById`) and the project has no jsdom/happy-dom environment configured; adding one just for this would be disproportionate. Verified in-browser instead: script injects/removes on toggle, events fire with correct data via a spied `window.umami`, and nothing throws when `window.umami` is absent.
- **`src/challenge.ts` (session 7) ‚Äî friend challenges ("beat my score" links), the app's deliberate viral loop.** After any Classic/Time Machine verdict, "‚öîÔ∏è Challenge a friend" packages the player's own batted innings as a URL: the matchup (batId/bowlId), book, era-adjust flag, and score travel in the URL **hash** (`/#challenge=v1.<base64url JSON>`) ‚Äî a hash and not a query string on purpose (never reaches the server or its logs, can't bust the service-worker cache). The friend opening it chases `runs + 1` under the identical odds; their verdict's "‚öîÔ∏è Fire one back" builds a counter-challenge from their own chase. Key invariants:
  - **`decodeChallenge` is deliberately paranoid** ‚Äî every field validated against roster/engine bounds (real roster ids in the correct roles so `.batting!`/`.bowling!` never see null; pages/runs/wickets/balls clamped; stats mode force-overrides the book to `VIRTUAL_BOOK`). Any anomaly ‚Üí `null` (the link just doesn't land), never a crash. Display-only ball tokens soft-fail to `[]` instead. **No names travel in the link, ever** ‚Äî identity comes from the chat app it's shared through; don't add a name field.
  - **The chase is live RNG** ‚Äî only the target travels, no seed. Structurally a challenge is a daily-style chase: `startChallenge` sets `innings = 2` from ball one, `state.challenge` non-null, and `playerBatting()`/`currentPair()` treat it exactly like `state.daily`. The challenger's innings arrives as tokens with **no luck data** ‚Äî `state.inn1.luck` is `[]`, which is why `recordFinishedMatch` guards `!state.challenge` before touching `inn1.luck[i]` (it would crash) and the challenge verdict shows a chase-only luck line instead of `luckReportHtml()`.
  - **Retries are allowed on purpose** ("took me five goes" is half the fun of a dare); every attempt records as a real match, `challengesWon` increments per winning chase. `pendingChallenge` lives at module level (not in `State`) so `freshSetup`/go-home can't drop an unaccepted dare off the home screen; it's never persisted ‚Äî re-opening the link recreates it.
  - The codec and validation are fully unit-tested (`tests/challenge.test.ts`, includes tamper tests that surgically rewrite wire fields); the DOM flow (landing card, verdict, hashchange) is untested-by-design like the rest of `main.ts` and verified in-browser.
- **Responsive play screen (session 8)** ‚Äî the play loop (decide ‚Üí flip ‚Üí observe) must fit one viewport with no scrolling, at both a 13" laptop (~1280√ó700) and a phone (375√ó812). `playHtml()` in `main.ts` wraps its markup in `.play-left` (book, flip card, commentary, ball log) and `.play-right` (banner, score bug, `.play-controls` ‚Üí stance/pp/ai-intent/flip button/hints, odds panel) ‚Äî written **`.play-right` first, then `.play-left`** in the template literal. That's deliberate ‚Äî it means the default (700‚Äì899px) single-column stack reproduces the original banner‚Üíscore‚Üíbook‚Üícontrols order for free, with zero JS/markup branching, just by writing `.play-right`'s content before `.play-left`'s.

  - **Dev-server gotcha hit while building this**: the `book-cricket-pwa` launch config runs `serve:pwa` (`vite build && vite preview`) ‚Äî a **static** file server over `dist/`, not a dev server with HMR. Editing CSS/TS and reloading the browser tab does nothing until you re-run `npm run build`; worse, the registered service worker (see the PWA bullet below) will keep serving an even-older cached build across reloads unless you unregister it and clear caches first. When verifying layout changes in-browser: rebuild, then unregister SW + clear caches, then hard-navigate, in that order, every time ‚Äî a `getComputedStyle(...)` sanity check on a value you just changed catches a stale build immediately if you skip a step.
- **FIRM RULE ‚Äî display name vs wire name (session 10, read this before renaming anything in this app).** "Stats mode" is called **Time Machine** on every user-facing surface (the home shelf card, the setup tab, the Play Store listing, the how-to panel) ‚Äî but `Mode = 'classic' | 'stats'` in `types.ts`, `state.mode === 'stats'`, the `data-action="mode-stats"`/`nav-stats"` attributes, the `.book-stats` CSS class, every engine parameter, and the `mode` value inside `match_started`/`match_finished` analytics events are **all still literally `'stats'`, permanently**. This was a deliberate session-10 decision, not an oversight: the Daily's determinism contract and the existing Umami event history both key off the string `'stats'`, and renaming a wire value mid-flight would silently fork the analytics data. **Any future rename request is a display-string change only** ‚Äî grep for the literal user-visible text (button labels, headings, disclaimers, `shareText()`), never touch an identifier, a `data-action`, a CSS class, or anything passed to `track()`. If a rename ever needs the *wire* value to change too (e.g. a genuine mode split), that is a breaking migration requiring its own explicit sign-off, not a documentation-driven cleanup.
- **Time Machine front door (session 10)** ‚Äî Riaan's-dad worked out that the app's best mode had its worst setup screen: Classic starts in one tap (a prefilled random book + flavor players), but Time Machine used to force two decisions across ~28 dense cards with nothing preselected and Start disabled until both were made. - **Hold-and-release flip (session 6)** ‚Äî the flip is a small state machine in `main.ts` (`flipPhase: idle | riffling | revealing`), not a one-shot. The book riffles continuously (`riffleTick`, a self-scheduling loop at `FLIP_RIFFLE_MS`, each frame a throttled `playFlip()`) and **the player decides when it lands**, three interchangeable ways: (1) **hold and release** ‚Äî press past `FLIP_TAP_MS` and let go (`stopFlip` on `pointerup`/`keyup`), natural on touch; (2) **tap to start, tap to stop** ‚Äî a quick press leaves it spinning (`flipStopArmed`, hint flips to "Tap to stop!"), the next press stops it, natural on a laptop/trackpad where holding a button is awkward; (3) a quick tap with **no follow-up auto-lands** after `FLIP_AUTOSTOP_MS` so nothing hangs and passive players still get a ball. `FLIP_MAX_HOLD_MS` is a safety cap. **The outcome is drawn in `drawCurrentBall()` at STOP, not at press** ‚Äî the release genuinely picks the ball from the (unchanged) odds, so the control is real rather than theatre. That's also why stance/power-play are **locked while `flipInProgress()`** (the `handleClick` guards) ‚Äî a mid-spin intent change must not leak into a draw that happens at release. Wiring: delegated `pointerdown/up/cancel` + `keydown/keyup` on `app` targeting `#flip-btn` (there is no `data-action="flip"` click path); `pointerdown` calls `preventDefault()` + `setPointerCapture`, `keydown` guards `e.repeat`. Was direct feedback from a non-cricketer playtester that one-tap flipping felt passive/low-agency. **CSS gotcha**: `#flip-btn` needs `touch-action: none` + `user-select/-webkit-touch-callout: none` so a hold doesn't start a scroll or pop the iOS selection callout ‚Äî don't remove those.

- **`reduceMotion` preference persistence (session 6 follow-up)** ‚Äî `state.reduceMotion` used to be computed fresh every `freshSetup()` from `prefersReducedMotion` (the OS-level `prefers-reduced-motion` media query) with no way for a player's explicit checkbox toggle to survive a reload ‚Äî a real bug, not just a UX rough edge: a player who unticked "Reduce animations" to fix their flip had that override silently discarded the next time they opened the app, since the OS signal reasserted itself every fresh load. - **`src/voiceLines.ts` / `src/commentators.ts` / `src/voice.ts`** ‚Äî pre-generated commentary, voiced on wickets, fours, sixes, power plays and verdicts only (never singles/twos/threes/fives ‚Äî see `resolveBallMoment`'s doc comment). `voiceLines.ts` is name-free script text (dynamic names stay in `commentary.ts`'s on-screen text). `commentators.ts` defines four **original commentator personas** (Enthusiast, Deadpan, Analyst, Showman) ‚Äî each an ElevenLabs stock/library voice pushed toward a character via `voiceSettings`, never a clone or impersonation of a real, named commentator; all four personas read the identical script, so the character comes entirely from the voice actor and delivery tuning. **Do not add a persona built from a real person's cloned voice** ‚Äî this was explicitly requested and declined (consent/right-of-publicity risk, worse for anyone deceased); stock/licensed library voices with original character framing are the line. `voice.ts` has two layers: pure resolvers (`resolveBallMoment`, `resolveMatchMoment`, unit-tested, and deliberately mirror `commentaryFor`'s priority order ‚Äî a `tests/voice.test.ts` case cross-checks the two never disagree) and untested browser playback (`playMomentVoice`/`playNameCallout`, both take a `personaId`) that fails silently on a 404. **A plain six has a 35% chance of playing just the batsman's name instead of a full line** (`revealBall` in `main.ts`) ‚Äî that's realistic (commentators really do just shout a name on a huge six). **A wicket never does this** ‚Äî real commentary always signals the dismissal itself ("OUT!", "Gone!"), so a bare bowler's name with nothing else doesn't read as a wicket; wickets always get a full reaction line. ## Product constraints

- Stats mode is a playful simulation, never a prediction ‚Äî the UI states this disclaimer in multiple places (setup panel, play screen, verdict, footer). Preserve it when touching those surfaces.
- `computeProbabilities` is surfaced verbatim in the in-game "How the odds work" panel; if you change its math, the explanation in `main.ts`'s `oddsBody()` must stay accurate. `oddsBody` renders a legible `<ul class="odds-mods">` of only the *active* factors (matchup, batter tempo from strike rate, settling-in, fatigue, era gap, stance, power play) above the raw percentage table ‚Äî not a wall of prose. When you add or change a modifier in the engine, add/adjust its line here so the panel doesn't quietly lie.
- The player bats innings 1 and every daily chase; the AI rival bats innings 2 of regular matches. Stance buttons and the power-play toggle must never render during the rival's innings ‚Äî the rival's intent line (`aiIntentText`) is the honest mirror of the same pure functions the flip will use.
- **State your batting/bowling role explicitly, never make the player infer it from names (session 6).** Real feedback: a 10-year-old playtester didn't realize he was bowling in innings 2 ‚Äî the only signals were the matchup names and the absent stance buttons, both of which read fine to an adult who already knows the game but say nothing explicit to a kid. `playHtml`'s `innings-banner` now always states `You bat` / `You bowl` up front (driven by `playerBatting()`), and the fighter card matching the player's own role gets a `¬∑ you` suffix. Don't regress this back to name-only banners even if it reads as redundant to an adult ‚Äî the target audience spans a wide age range and the redundancy is the point.
- One power play per innings per side (`ppUsed` resets in `startChase`/`startSpell`/`startDaily`). A doubled ball stores the *drawn* outcome (`Ball.doubled` flag, `√ó2` token suffix); doubling is applied at scoring/display time so the page metaphor, luck pricing and share grid stay consistent.
- `#flip-btn` is `position: sticky; bottom: 14px` (session 5) ‚Äî it must stay reachable without scrolling on any screen height, and stay put as `#ball-log` grows a chip taller every single ball (without this it would slide further down the page each flip, not just at match start). Don't remove the sticky positioning to "simplify" the CSS.
- `.verdict` has `max-height: calc(100vh - 32px); overflow-y: auto` (session 5) ‚Äî a Gauntlet-conclusion verdict (series line + next-opponent teaser + record notes + luck report + disclaimer + 3 action buttons) is taller than most viewports, and `.overlay` itself doesn't scroll. Without the modal's own internal scroll, both the heading and the action buttons become completely unreachable ‚Äî confirmed by reproducing it at a 600px-tall viewport before the fix existed. Keep this whenever adding more content to any verdict.
- Respect the `reduceMotion` state flag: animations gate on it and durations drop to 0.
