# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Book Cricket Time Machine вАФ a browser game reviving Indian schoolyard book cricket (flip to a random page; last digit `0` = out, `1вАУ6` = that many runs, `7/8/9` = 1 run), with "what if" duels between real cricket legends. Entirely client-side: vanilla TypeScript + Vite, no framework, no backend, no accounts вАФ persistence is a single localStorage blob (`src/storage.ts`), nothing leaves the device. A match is two innings вАФ your XI bats, then the rival XI chases; each innings is a spell of 12 balls or 2 wickets (`SPELL` in `src/engine.ts`). The verdict includes a luck report built from per-ball probability snapshots (`expectedRuns`/`outcomeChance`; Classic balls are priced by `classicProbabilities`). The Daily Challenge is a seeded chase (`src/daily.ts`): the rival's innings, players and book are deterministic from the local date, so everyone in the world faces the same target; the player gets one attempt a day, a played-streak, and a Wordle-style emoji share grid.

## Commands

```bash
npm run dev        # Vite dev server
npm test           # vitest run (all tests)
npx vitest run tests/engine.test.ts -t "sums to 1"   # single test by name
npm run build      # tsc type-check + vite build
npm run preview    # serve the production build
npm run voice:generate  # render commentary MP3s via ElevenLabs (needs .env.local, see below)
```

There is no linter configured; `tsc` (strict mode, noUnusedLocals/Parameters) is the only static check and only runs via `npm run build`.

`voice:generate` runs `scripts/generate-voice-clips.ts` via `vite-node` (already a transitive dep of vitest вАФ no new package added) so it can import `src/voiceLines.ts`, `src/commentators.ts` and `src/roster.ts` directly as the single source of truth. It loops every persona in `COMMENTATORS` √Ч every line/name, using each persona's own `voiceId`/`voiceSettings` (not a single global voice). It reads `ELEVENLABS_API_KEY` (required) and optional `ELEVENLABS_MODEL_ID` from a hand-rolled `.env.local` loader (no dotenv dependency; the file is covered by the existing `*.local` gitignore pattern вАФ never paste the key into chat or commit it). It's idempotent вАФ reruns skip files that already exist вАФ and writes into `public/audio/voice/<personaId>/`. Those MP3s are committed like any other static asset (not gitignored): that keeps a git-integrated static host (Vercel/Netlify) able to build and deploy without ever needing the ElevenLabs key as a CI secret. Adding a fifth persona means adding one entry to `COMMENTATORS` and rerunning the script вАФ it only synthesizes the new persona's files, since existing ones are skipped.

## Architecture

Two-layer split вАФ pure logic vs. DOM вАФ is the core invariant:

- **`src/engine.ts`** вАФ all game logic as pure functions. Every random draw takes an injectable `Rng` (defaults to `Math.random`) so tests are deterministic. Session-2 additions: `STANCES` (defend/normal/attack multipliers on wicket odds and boundary weights), power play (runs double at scoring time; wicket odds √Ч2 capped at 60% in Stats, and in Classic the 7/8/9 mercy digits become OUT), the rival chase brain (`chaseStance`, `chaseUsesPowerPlay` вАФ pure, so the UI announces intent honestly before the flip), and Gauntlet seeding ratings. **All new `computeProbabilities` parameters default to exact identity** (`'normal'`, `false`) вАФ the daily challenge seeds through the default path, and `tests/daily.test.ts` pins Daily #1's full fingerprint as a regression tripwire. Two ball-drawing paths:
  - *Classic mode*: `drawClassic` picks a real page number; outcome follows from its last digit.
  - *Stats mode*: `drawOutcome` samples from `computeProbabilities(bat, bowl, eraAdjusted)`, then `pageForOutcome` reverse-engineers a page whose last digit would have produced that outcome вАФ the book metaphor is kept honest by construction, so don't break that pairing.
- **`src/main.ts`** вАФ the only file that touches the DOM. Single mutable `State` object with three phases (`home` вАФ the nostalgic pavilion/landing screen вАФ then `setup` and `play`); setup screen is re-rendered via full `innerHTML` replacement (`render()`), while play-screen ball updates and text inputs are patched by targeted DOM mutation to preserve animations and input focus (see `handleInput`'s "no re-render: don't steal focus" pattern). Events are delegated through `data-action` attributes on one root listener. A daily match is `state.daily !== null`: it skips innings 1 (pre-seeded), always chases, and its verdict has no "Play Again".
- **`src/daily.ts`** вАФ pure Daily Challenge logic: `mulberry32` seeded PRNG, local-date day keys (never `toISOString` вАФ the day rolls at local midnight), and `generateDaily`, which draws players, book and the rival's full innings off one seeded stream. **Draw order is part of the contract** вАФ reordering the draws (or editing `DAILY_BOOKS`, the roster order, or `computeProbabilities`) silently changes every past and future challenge. `DAILY_EPOCH_KEY` fixes Daily #1 and must never move once share grids are public. `MIN_DAILY_TARGET` (session 3) is a dud-day floor: if the simulated innings collapses below it, `generateDaily` re-simulates from the *same continuing* rng stream (never re-seeds) via `simulateInnings`, so a day that already clears the floor вАФ like Daily #1, target 20 вАФ never retries and stays byte-identical. `tests/daily.test.ts` sweeps thousands of day keys to confirm the floor holds; that sweep, plus the Daily #1 pin, must both keep passing before touching this function again.
- **`src/storage.ts`** вАФ the only module that may touch localStorage: one versioned blob (career ledger, daily streaks, luckiest-ever ball, sound pref) behind an injectable `Backing` so tests use a Map and private-browsing mode degrades to in-memory. `beginDailyAttempt` marks the day as played *when the chase starts*, so a mid-match refresh can't buy a second attempt.
- **`src/roster.ts`** вАФ data-only player list. Adding a player requires no simulation changes: bowlers get a `bowling` block, batsmen a `batting` block; all-rounders get both and appear in both pickers. Stats are deliberately approximate format blends tuned for fun.
- **`src/commentary.ts`** вАФ phrase templates with `{batsman}`/`{bowler}`/`{page}` placeholders (filled via `fillTemplate`) and a module-level `lastPhrase` guard so no phrase repeats on consecutive balls (tests rely on `resetCommentary()`).
- **`src/audio.ts`** вАФ synthesized Web Audio sound cues (flip/runs/boundary/wicket); no audio assets. The sound toggle persists via `storage.ts` like everything else. The footer promise is "your scorebook lives only on this device вАФ no accounts, no tracking": localStorage is fine, but nothing may ever be sent over the network.
- **`src/voiceLines.ts` / `src/commentators.ts` / `src/voice.ts`** вАФ pre-generated commentary, voiced on wickets, fours, sixes, power plays and verdicts only (never singles/twos/threes/fives вАФ see `resolveBallMoment`'s doc comment). `voiceLines.ts` is name-free script text (dynamic names stay in `commentary.ts`'s on-screen text). `commentators.ts` defines four **original commentator personas** (Enthusiast, Deadpan, Analyst, Showman) вАФ each an ElevenLabs stock/library voice pushed toward a character via `voiceSettings`, never a clone or impersonation of a real, named commentator; all four personas read the identical script, so the character comes entirely from the voice actor and delivery tuning. **Do not add a persona built from a real person's cloned voice** вАФ this was explicitly requested and declined (consent/right-of-publicity risk, worse for anyone deceased); stock/licensed library voices with original character framing are the line. `voice.ts` has two layers: pure resolvers (`resolveBallMoment`, `resolveMatchMoment`, unit-tested, and deliberately mirror `commentaryFor`'s priority order вАФ a `tests/voice.test.ts` case cross-checks the two never disagree) and untested browser playback (`playMomentVoice`/`playNameCallout`, both take a `personaId`) that fails silently on a 404. **The game must be fully playable with zero audio files present** вАФ clips are static MP3s generated by `scripts/generate-voice-clips.ts` (`npm run voice:generate`, needs `ELEVENLABS_API_KEY` in a gitignored `.env.local`; never call this API from the shipped app itself, or the key would leak to every player), one set per persona under `public/audio/voice/<personaId>/`.
- **`src/avatar.ts`** вАФ generated inline-SVG avatars (initials + emoji + country gradient); no photo assets.
- **PWA shell (session 3)** вАФ `public/manifest.webmanifest`, `public/icons/*.png` (generated once via a throwaway `sharp` install вАФ `npm install --no-save sharp`, never a persisted dependency вАФ from the SVG source described in the README; regenerate the same way if the mark changes), and `public/sw.js`, a hand-rolled service worker (no build plugin, no new dependency): cache-first-with-background-refresh for same-origin GET requests, a small explicit precache list (`/`, `/manifest.webmanifest`), registered unconditionally in `main.ts`. Verify PWA behavior against `npm run serve:pwa` (builds, then serves `dist/` on :4300) вАФ the raw `npm run dev` server also registers the SW, which can cause stale-content confusion while iterating; unregister it in devtools if that happens.
- **`shareOrCopy` in `main.ts`** (session 3) вАФ tries `navigator.share` first (native share sheet, mobile), swallows `AbortError` (user cancelled) silently, falls back to the existing `copyToClipboard` for everything else including desktop browsers without the API. `CAN_SHARE` is detected once at module load, driving the button label (`рЯУ§ Share вА¶` vs `рЯУЛ Copy вА¶`).

Tests (`tests/engine.test.ts`, `tests/daily.test.ts`, `tests/storage.test.ts`, `tests/voice.test.ts`, `tests/commentators.test.ts`, vitest) cover only the pure layer вАФ engine math, commentary non-repetition, daily determinism, storage recorders, voice-moment resolution, and the persona manifest's sanity (unique ids, valid `voiceSettings` ranges). Keep new game logic in `engine.ts`/`daily.ts`/`voice.ts`'s pure half so it stays testable; `main.ts` (and `voice.ts`'s playback half) is untested by design.

`design-system/` holds standalone HTML preview cards (tokens, components) for the claude.ai/design pane, each tagged with a `@dsCard` marker вАФ they are reference artifacts, not part of the build.

## Product constraints

- Stats mode is a playful simulation, never a prediction вАФ the UI states this disclaimer in multiple places (setup panel, play screen, verdict, footer). Preserve it when touching those surfaces.
- `computeProbabilities` is surfaced verbatim in the in-game "How the odds work" panel; if you change its math, the explanation text in `main.ts`'s `oddsPanel()` must stay accurate вАФ including the stance/power-play multiplier lines in `oddsBody()`.
- The player bats innings 1 and every daily chase; the AI rival bats innings 2 of regular matches. Stance buttons and the power-play toggle must never render during the rival's innings вАФ the rival's intent line (`aiIntentText`) is the honest mirror of the same pure functions the flip will use.
- One power play per innings per side (`ppUsed` resets in `startChase`/`startSpell`/`startDaily`). A doubled ball stores the *drawn* outcome (`Ball.doubled` flag, `√Ч2` token suffix); doubling is applied at scoring/display time so the page metaphor, luck pricing and share grid stay consistent.
- Respect the `reduceMotion` state flag: animations gate on it and durations drop to 0.

## Deployment

Static site, zero backend вАФ `vercel.json`/`netlify.toml` both just point at `npm run build` вЖТ `dist/`. Once deployed, set `SHARE_URL` in `src/daily.ts` (currently `''`) to the live domain so share text includes a link, and make the `og:image`/`twitter:image` paths in `index.html` absolute (they're relative for now, since the domain isn't known yet).
